## 2. قانون اول — `unicode-bidi: plaintext` برای بخش‌های فنی داخل متن مخلوط

**مشکل:** وقتی یک رشته‌ی فنی (آدرس، Path، نام فایل، یک عبارت انگلیسی) وسط یک جمله‌ی فارسی RTL
قرار می‌گیرد، الگوریتم BiDi مرورگر می‌تواند جای کاراکترهای آن رشته یا علائم نگارشی مجاورش
(`:`، `/`، `.`) را به‌طور غیرمنتظره به‌هم بریزد — چون مرورگر سعی می‌کند کل خط را به‌صورت یک‌دست
RTL/LTR تفسیر کند.

**قانون:** هر بخش فنی که داخل یک بلوک متن RTL embed می‌شود (یک مقدار، نه یک پاراگراف کامل
انگلیسی)، باید با `unicode-bidi: plaintext` + `direction: ltr` ایزوله شود تا BiDi Algorithm
مرورگر آن را به‌عنوان یک واحد مستقل LTR در نظر بگیرد، نه بخشی از جریان RTL اطرافش.

```css
.tech-fragment {
  unicode-bidi: plaintext;
  direction: ltr;
}
```

```html
<p>سرور با آدرس <span class="tech-fragment">192.168.1.1:8080</span> پاسخ نداد.</p>
```

> این قانون فقط برای **بخش‌های کوتاه و فنی** است (آدرس، UUID، Path، نام پروتکل) — نه برای کل
> یک بلوک متنی که خودش زبان دیگری دارد (برای آن `dir="auto"` کافی است؛ قانون ۳ همین سند پایین).

## 3. قانون دوم — `<bdi>` دور هر عدد چندرقمی/کد داخل جریان RTL

**مشکل:** اعدادِ چندرقمی (Port، شمارنده، شناسه‌ی عددی، هر Code) وقتی مستقیماً وسط متن RTL رندر
می‌شوند، می‌توانند توسط BiDi Algorithm به‌اشتباه «معکوس» در نظر گرفته شوند — این یک کلاس باگ
واقعی و سندشده در پروژه‌های دوزبانه‌ی مشابه است (مخصوصاً وقتی عدد بعد از یک کاراکتر RTL و قبل از
یک علامت نگارشی RTL می‌آید).

**قانون:** هر عدد چندرقمی یا کدی که داخل یک جریان متنی RTL رندر می‌شود، با تگ بومی HTML `<bdi>`
(Bidirectional Isolate) پوشانده شود — نه با یک `<span dir="ltr">` دستی، چون `<bdi>` دقیقاً برای
همین مورد طراحی شده و بدون نیاز به دانستن جهت محتوای داخلی، آن را ایزوله می‌کند.

```html
<p>درگاه <bdi>8443</bdi> در حال استفاده توسط <bdi>3</bdi> نود دیگر است.</p>
```

> طبق بخش ۹.۵ سند `07-UI_UX_SYSTEM` (مرز اعداد)، خودِ رقم‌ها همیشه ASCII باقی می‌مانند — `<bdi>`
> فقط **ترتیب نمایش** آن‌ها را در جریان RTL درست نگه می‌دارد، کاری به نوع رقم (فارسی/ASCII) ندارد.

## 4. قانون سوم — CSS Logical Properties به‌جای Physical Properties (بدون استثنا)

**مشکل:** `margin-left`/`margin-right`، `padding-left`/`padding-right`، `left`/`right` (Physical
Properties) جهت **فیزیکی** صفحه را فرض می‌گیرند؛ در یک صفحه‌ی RTL همه‌ی این مقادیر باید دستی
Mirror شوند، که در عمل یا فراموش می‌شود یا با CSS تکراری/Override شده برای حالت RTL مدیریت
می‌شود (Code Duplication دائمی).

**قانون:** در **هر CSS جدید، بدون استثنا**، از CSS Logical Properties استفاده شود — این مقادیر
خودشان بر اساس `dir` فعال (`rtl`/`ltr`) جهت درست را می‌گیرند، بدون نیاز به هیچ Override جدا برای
RTL.

| به‌جای (Physical) | استفاده شود (Logical) |
|---|---|
| `margin-left` / `margin-right` | `margin-inline-start` / `margin-inline-end` |
| `padding-left` / `padding-right` | `padding-inline-start` / `padding-inline-end` |
| `left` / `right` | `inset-inline-start` / `inset-inline-end` |
| `border-left` / `border-right` | `border-inline-start` / `border-inline-end` |
| `text-align: left/right` | `text-align: start` / `text-align: end` |

```css
/* ❌ بد — در RTL باید دستی Mirror شود */
.sidebar-item {
  margin-left: 12px;
  border-right: 1px solid var(--border-color);
}

/* ✅ خوب — خودش با dir فعال هماهنگ می‌شود */
.sidebar-item {
  margin-inline-start: 12px;
  border-inline-end: 1px solid var(--border-color);
}
```

> **بدون استثنا** یعنی بدون استثنا: حتی برای یک فاصله‌ی ظاهراً بی‌اهمیت یا یک Component که فعلاً
> «فقط در حالت انگلیسی» دیده می‌شود — چون طبق سند ۰۷ §۹.۱۰، پولیش کامل RTL به فاز نهایی موکول
> شده و هر Physical Property که الان نوشته شود، در آن فاز باید پیدا و اصلاح شود؛ نوشتن درست از
> همین الان آن بدهی را از ریشه حذف می‌کند.

## 5. جمع‌بندی Checklist (قبل از هر Commit مرتبط با UI)

- [ ] هیچ `margin-left/right`، `padding-left/right`، `left/right` فیزیکی جدید اضافه نشده؛ همه
      Logical Properties هستند (بخش ۴).
- [ ] هر عدد چندرقمی/کد که داخل متن RTL رندر می‌شود، با `<bdi>` پوشیده شده (بخش ۳).
- [ ] هر بخش فنی Embed‌شده داخل یک جمله‌ی RTL، `unicode-bidi: plaintext` + `direction: ltr` دارد
      (بخش ۲).
- [ ] فیلدهای محتوای آزاد کاربر (مثل Remark) مستقل `dir="auto"` دارند (سند ۰۷ §۹.۸) — جدا از
      جهت کل صفحه.
- [ ] هیچ رقم فارسی/عربی در هیچ متن رندرشده‌ای نیست؛ اعداد همیشه ASCII هستند (سند ۰۷ §۹.۵).
- [ ] این سند، همین الان، قبل از این Commit، یک‌بار کامل خوانده شده است.
