## ۱. چرا الان، و چرا این تصمیم نمی‌تواند دوباره عقب بیفتد

این تنش از همان روزهای اول پروژه ثبت شده و عمداً عقب افتاده («این تصمیم باید قبل از Phase 9 گرفته
شود، نه همین الان» — `IMPLEMENTATION_BLUEPRINT` §4). آن «همین الان» همان لحظه‌ای بود که فاز ۱
شروع می‌شد؛ الان Phase 8 تمام شده و Phase 9 دقیقاً همان فازی است که باید قبلش حل شود. عقب‌انداختن
بیشتر یعنی شروع کدنویسی UI با یک فرض ضمنی (مثلاً Zero-Build) که اگر بعداً غلط از آب در بیاید،
بازنویسی همه‌ی Componentها لازم می‌شود — دقیقاً نوع هزینه‌ای که قانون ADR (Rule 13) می‌خواهد قبل از
وقوع شناسایی شود، نه بعدش.

**عدد واقعی امروز:** `core/` همین الان ۸۵ فایل `.js` دارد (Phase 1 تا 8)، همه Native ESM + JSDoc،
بدون هیچ Bundler. `ui/` هنوز کاملاً خالی است (فقط ۸ پوشه‌ی `.gitkeep`). یعنی هر تصمیمی که الان
گرفته شود، **هیچ هزینه‌ی Migration روی Core ندارد** — این تصمیم فقط مسیر آینده‌ی `ui/` و فایل
نهایی Packaging را تعیین می‌کند، نه چیزی که از قبل نوشته شده. این فرصت را داریم چون «الان» واقعاً
ارزان‌ترین لحظه‌ی ممکن برای این تصمیم است.

## ۲. خود تنش (دقیقاً طبق سند)

دو قانون قفل‌شده با هم در تناقضند:

1. **`No Build Step`** (`01-MASTER_BLUEPRINT` / `14-DEPENDENCY_POLICY`) — اپ باید بدون هیچ مرحله‌ی
   Compile/Bundle، با باز کردن مستقیم HTML اجرا شود.
2. **خروجی نهایی = «Single HTML Application»** (`IMPLEMENTATION_BLUEPRINT` §4) — یک فایل، نه یک
   پوشه‌ی چند-ده-فایلی.

این دو با حجم واقعی پروژه (ده‌ها ماژول طبق `MASTER_FILE_STRUCTURE`) در تنش‌اند: راه‌حل بدیهی برای
«همه را در یک فایل جمع کن» یک Bundler است (esbuild و مشابه)، ولی همین Bundler یعنی یک Build Step —
دقیقاً همان چیزی که قانون (۱) ممنوع کرده.

یک تنش دوم به همین گره وصل است (طبق بازبینی خود سند، «یک سؤال واحد»، نه دو ADR جدا):
`htm` (جایگزین JSX بدون Build، تأییدشده در `14-DEPENDENCY_POLICY` همراه Preact) با TypeScript
خوب کار نمی‌کند (Tagged Template، Type Inference ضعیف). یعنی اگر TypeScript روزی از «Optional
Future» به نیاز واقعی تبدیل شود، دقیقاً همین سؤال Build Step دوباره مطرح می‌شود.

سؤال واحد: **آیا یک Build Step سبک (فقط برای Bundle/Type-Check نهایی، نه برای حلقه‌ی توسعه‌ی
روزانه) پذیرفته می‌شود یا نه؟**

### یک ریسک عملی که سند هم اشاره کرده ولی ارزش تأکید مجدد دارد
مرورگرهای مبتنی بر Chromium (و WebViewهای اندرویدی مشابه) `import` ماژول‌های ES را از پروتکل
`file://` با خطای CORS رد می‌کنند («Cross origin requests are only supported for
http/https/...»). این یعنی **حالت ساده‌ترین Deployment Mode پروژه («Mode 1 — Open HTML File
مستقیم»، `IMPLEMENTATION_BLUEPRINT` §5)** با معماری Native-ESM-چندفایلی به‌تنهایی درست کار
نمی‌کند — مگر همه‌چیز واقعاً در یک `<script>` Inline باشد (نه `import` بین فایل‌ها)، یا کاربر آن
را از یک Local Server باز کند (که دیگر «باز کردن مستقیم فایل» نیست). هر گزینه‌ی پایین این نکته را
باید صریح حل کند.

### نکته‌ی Worker (مرتبط، نه جدا)
`core/worker/worker-manager.js` همین الان با تزریق `workerFactory` نوشته شده — یعنی *چطور* Worker
واقعی ساخته می‌شود (`new Worker(url, {type:'module'})` روی یک فایل جدا، یا از Blob URL، یا Inline)
عمداً به بیرون از Core موکول شده. این تصمیم خوب بوده، ولی یعنی این Gap هنوز باز است: یک Single-HTML
File نمی‌تواند یک فایل Worker جدا Load کند؛ `ULTIMATE_BLUEPRINT_INDEX` همین را به‌عنوان بخشی از
همین Gap ثبت کرده («نحوه‌ی Inline کردن Workerها»). هر گزینه باید پاسخی برای این هم داشته باشد.

## ۳. گزینه‌ها

### گزینه ۱ — Zero-Build کامل (تا همیشه)

JavaScript خالص + `htm` + Native ES Modules. نه TypeScript، نه Bundler — نه در توسعه، نه در
خروجی نهایی.

**چطور با تنش‌ها کنار می‌آید:**
- خروجی «Single HTML» با Native ESM چندفایلی همخوان نیست؛ راه‌حل واقعی در این گزینه فقط می‌تواند
  «همه‌چیز را دستی/با یک اسکریپت ساده‌ی Concatenation (نه Transpile، فقط چسباندن متن فایل‌ها) در
  یک `<script>` بریزیم» باشد — که خودش یک نوع Build Step بدوی است، فقط بدون Compiler.
- Worker Inlining: باید Worker Source را هم به‌صورت رشته Inline کرد (مثلاً با Blob URL از یک
  Template String) — دستی و نسبتاً شکننده در نگه‌داری.

**پیامدها:**
| جنبه | اثر |
|---|---|
| همخوانی با قانون قفل‌شده No-Build-Step | ✅ کامل — هیچ Compiler در هیچ مرحله |
| TypeScript | ❌ برای همیشه «Future» می‌ماند مگر بعداً با ADR جدید عوض شود |
| تجربه‌ی توسعه (Dev Loop) | ✅ ساده‌ترین — Refresh مرورگر، بدون انتظار Build |
| ریسک file:// CORS (Mode 1) | 🔴 باقی می‌ماند مگر Concatenation دستی/اسکریپتی انجام شود |
| Worker در فایل نهایی | 🟡 فقط با ترفند Inline/Blob URL دستی، نگه‌داری شکننده‌تر |
| هزینه‌ی Migration روی Core فعلی | ✅ صفر — Core همین الان همین‌طور است |
| Bundle Size Budget (سند 14 §2.1) | بدون Minify خودکار، رسیدن به سقف gzip سخت‌تر کنترل می‌شود (باید دستی/با ابزار خارج از Pipeline سنجیده شود) |
| ریسک بلندمدت | اگر نیاز واقعی به TypeScript پیدا شود، این تصمیم باید کامل دوباره باز شود |

### گزینه ۲ — Build Step سبک، فقط برای Bundle/Type-Check نهایی

پذیرش یک Build Step که **در حلقه‌ی توسعه‌ی روزانه دخالت نمی‌کند** (در Dev، همان Native ESM مستقیم
از فایل‌ها سرو می‌شود) ولی برای ساخت artifact نهایی (Single HTML + Type-Check) استفاده می‌شود.
TypeScript + JSX از همان ابتدا برای کد جدید (`ui/`). نیاز به اصلاح رسمی قانون No-Build-Step با یک
ADR.

**نکته‌ی مهم که سند صراحتاً نگفته ولی از وضعیت واقعی پروژه روشن است:** این گزینه لازم نیست
۸۵ فایل فعلی `core/` را به TypeScript واقعی (`.ts`) مهاجرت دهد. `core/` همین الان JS + JSDoc +
`tsc --noEmit` Type-Checked است (دقیقاً همان مدل فاز ۱)؛ Build Step می‌تواند فقط روی **خروجی
نهایی** (Bundle کردن `ui/` + Core + تولید یک فایل HTML) کار کند و کدِ منبع Core دست‌نخورده، بدون
هیچ مرحله‌ی Compile، باقی بماند. یعنی این گزینه را می‌توان به‌صورت محدود («فقط `ui/` و فقط
Assembly نهایی») هم اجرا کرد، نه لزوماً TypeScript-سازی کل پروژه.

ابزار طبیعی برای این مدل (Native ESM در Dev، Bundle واقعی فقط برای Production) دقیقاً همان چیزی
است که Vite برایش ساخته شده؛ esbuild (که خود `IMPLEMENTATION_BLUEPRINT` §4 به‌عنوان نمونه‌ی
Bundler نام برده) هم به‌تنهایی یا زیرِ Vite قابل استفاده است.

**پیامدها:**
| جنبه | اثر |
|---|---|
| همخوانی با قانون قفل‌شده No-Build-Step | ❌ نیاز به اصلاح رسمی قانون با ADR (دقیقاً همان «استثنای Preact» در سند 14، یک الگوی از‌قبل‌پذیرفته‌شده برای این نوع اصلاح) |
| TypeScript | ✅ از روز اول `ui/` ممکن است؛ `htm` دیگر لازم نیست (JSX واقعی) |
| تجربه‌ی توسعه (Dev Loop) | ✅ اگر Vite/مشابه: تقریباً به‌اندازه‌ی Zero-Build سریع (Native ESM در Dev) |
| ریسک file:// CORS (Mode 1) | ✅ حل‌شده — Bundler خروجی نهایی را به یک `<script>` Inline تبدیل می‌کند |
| Worker در فایل نهایی | ✅ حل‌شده — اکثر Bundlerها (esbuild/Vite/Rollup) از Worker Inlining (Blob/Data URL) به‌صورت توکار پشتیبانی می‌کنند |
| هزینه‌ی Migration روی Core فعلی | ✅ صفر اگر محدود به `ui/` + Assembly بماند (طبق نکته‌ی بالا) |
| Bundle Size Budget (سند 14 §2.1) | ✅ Minify/Tree-Shaking خودکار، کنترل سقف ۵۰/۱۵۰KB ساده‌تر |
| هزینه‌ی جدید | یک Dev Dependency جدید (Vite/esbuild) نیاز به همان فرآیند «Future Dependency Review» سند 14 §6 دارد؛ Pipeline CI/Release یک مرحله‌ی جدید (Build) پیدا می‌کند |

### گزینه ۳ — توزیع پوشه‌ای برای PWA/Capacitor؛ Single-File فقط برای Mode «باز کردن مستقیم»

به‌جای یک تصمیم واحد برای همه‌ی Deployment Modeها، تصمیم را بر اساس Mode تفکیک کن:
- **Mode 1 (Open HTML File مستقیم):** همچنان Zero-Build، ولی محدوده‌ی این Mode (و انتظارات از
  آن) رسماً کوچک‌تر تعریف می‌شود — یا با Concatenation دستی همان مشکل گزینه ۱ حل می‌شود، یا این
  Mode از همان ابتدا به‌عنوان «نسخه‌ی ساده‌تر/محدودتر» شناخته می‌شود.
- **Mode 2-4 (PWA / TWA / Capacitor):** این Modeها ذاتاً پوشه‌ای هستند (Capacitor و TWA به یک
  دایرکتوری Asset نیاز دارند، نه یک فایل) و همیشه روی `https://` یا یک Origin مشابه (نه `file://`)
  سرو می‌شوند — یعنی مشکل CORS اصلاً برایشان وجود ندارد، و Build Step اینجا طبیعی‌تر است.

**پیامدها:**
| جنبه | اثر |
|---|---|
| همخوانی با قانون قفل‌شده No-Build-Step | 🟡 جزئی — Mode 1 همچنان Zero-Build می‌ماند؛ Modeهای دیگر از قانون مستثنا می‌شوند (نیاز به ADR برای این استثنا، شبیه استثنای Preact) |
| TypeScript | ✅ برای Mode 2-4 ممکن؛ Mode 1 همچنان محدود به همان محدودیت‌های گزینه ۱ |
| تجربه‌ی توسعه (Dev Loop) | دو مسیر موازی (یکی برای Mode 1، یکی برای 2-4) — پیچیدگی نگه‌داری بیشتر از گزینه‌های ۱/۲ |
| ریسک file:// CORS | فقط برای Mode 1 باقی می‌ماند (با همان راه‌حل‌های گزینه ۱) |
| Worker در فایل نهایی | برای Mode 2-4 ساده (فایل جدا، Origin واقعی)؛ برای Mode 1 همچنان چالش گزینه ۱ |
| هزینه‌ی Migration روی Core فعلی | ✅ صفر |
| ریسک | دو Pipeline یعنی دو جا برای باگ مخفی شدن (چیزی که در Mode 1 کار می‌کند ممکن است در Mode 4 فرق کند یا برعکس) — نیاز به تست هر دو مسیر در 15-TESTING_FRAMEWORK |

## ۴. نکات تصمیم‌ساز (نه تصمیم — فقط برای کمک به انتخاب)

- اگر **اولویت واقعی همان P1 سند `IMPLEMENTATION_BLUEPRINT` (Android, مخصوصاً Termux/مستقیم-فایل)**
  باشد و TypeScript واقعاً «Future» بماند، گزینه‌ی ۱ کمترین تصمیم جدید را تحمیل می‌کند — ولی ریسک
  CORS باید همان حالا با یک راه‌حل مشخص (نه «بعداً حل می‌کنیم») جواب داده شود.
- اگر **TypeScript به‌زودی واقعاً لازم می‌شود** (نه فقط «Optional Future» تئوریک)، گزینه‌ی ۲
  کم‌اصطکاک‌ترین مسیر است، چون طبق بررسی بالا هیچ هزینه‌ی Migration روی ۸۵ فایل فعلی `core/` ندارد —
  فقط `ui/` (که هنوز خالی است) را تحت تأثیر قرار می‌دهد.
- اگر **چهار Deployment Mode به‌طور واقعی همه اولویت یکسان دارند** (نه فقط Mode 1)، گزینه‌ی ۳
  منطقی‌تر است، ولی هزینه‌ی نگه‌داری دو Pipeline را باید پذیرفت.
- هرکدام انتخاب شود، طبق `ULTIMATE_BLUEPRINT_INDEX` («Gap شناخته‌شده — Build & Bundling
  Strategy») خروجی باید یک **Full ADR** (شماره‌ی بعدی: `ADR-014`) + یک سند/بخش مشخص‌کننده‌ی
  Bundler انتخابی، روش Inline کردن Worker، و Minify/Inject شدن CSS/JS باشد.

## ۵. خارج از این تصمیم

این پیشنهاد فقط مسیر Packaging/Build را مشخص می‌کند. تصمیم Preact (ADR-001) و تصمیم Vitest/JS+JSDoc
برای Phase 1 (ADR-005) باز هستند و توسط این پیشنهاد تغییر نمی‌کنند — این فقط درباره‌ی *چطور به یک
فایل نهایی می‌رسیم*، نه *با چه ابزاری UI می‌نویسیم* یا *چطور Core را تست می‌کنیم*.

**هیچ کد UI/Preact یا Component‌ای تا تأیید این تصمیم توسط مهدی نوشته نمی‌شود.**
