# ANTI-CHAOS BLUEPRINT

**Universal Network Config Toolkit (UNCT)**

| | |
|---|---|
| **Status** | LOCKED — بدون تغییر منطقی، فقط بازآرایی |
| **Priority** | CRITICAL — قوانین حاکم بر کل پروژه |

> این سند ۱۰ قانون مطلق پروژه است. این قوانین مستقل از هر تصمیم فنی (مثل انتخاب Preact یا هر کتابخانه‌ی دیگر) هستند و هرگز نقض نمی‌شوند.

### مرز دقیق Rule 11 — Selector Pattern *(جدید — بازبینی نهایی)*

> ⚠️ **گلوگاه واقعی:** مرز Rule 11 برای حالت‌های ساده (UI که خودش UUID را Parse کند) واضح است، ولی برای حالت‌های مرزی روشن نبود: اگر UI بخواهد بر اساس `securityScore` (که از قبل توسط Analyzer محاسبه شده) فیلتر یا مرتب‌سازی کند، آیا این «منطق Core» محسوب می‌شود؟

**قانون رسمی:** UI هرگز مستقیماً از `core/store/` (Adapter Layer، سند 01) Raw UNM نمی‌خواند؛ بلکه از طریق **Selectorها** (توابع کوچک و خالص که یک View Model آماده برمی‌گردانند، مثل `selectNodesSortedBySecurity()`) داده می‌گیرد.

| عملیات | مجاز در UI (از طریق Selector)؟ |
|---|---|
| فیلتر/مرتب‌سازی بر اساس مقداری که **از قبل توسط Core محاسبه شده** (مثل `securityScore`) | ✅ مجاز — این نمایش داده است، نه تولید آن |
| محاسبه‌ی یک امتیاز یا اعتبارسنجی جدید داخل UI | ❌ ممنوع — این منطق Core است و باید در Analyzer/Validator باشد |

> Selectorها در `core/store/` یا یک پوشه‌ی مجزا (مثل `ui/selectors/`) قرار می‌گیرند، نه داخل خود Componentهای Preact — تا UI Component فقط «نمایش» باقی بماند.

## یادآوری کاربردی (بدون تغییر در خود قوانین)

این قوانین در طول بازنویسی بلوپرینت‌ها به‌صورت عملی رعایت شده‌اند، برای مثال:

- Rule 07 → دلیل اینکه ایده‌های Future/Backlog (در اسناد 03، 07، 08) حذف نشدند، فقط دسته‌بندی شدند.
- Rule 08 → دلیل اینکه تصمیم Preact (اسناد 09، 11، 13، 14) با یک جدول معیار مقایسه‌ای مستند شد، نه یک تصمیم بی‌دلیل.
- Rule 10 → دلیل اینکه سند 05 (UNM) اولین سندی بود که دقیق‌سازی شد.
